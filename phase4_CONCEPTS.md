# Phase 4 — FastAPI + Streamlit Cheat Sheet

## Architecture

```
┌──────────────────┐         HTTP / SSE          ┌───────────────────┐
│  Streamlit UI     │ ───────────────────────────► │  FastAPI Backend   │
│  (port 8501)       │ ◄─────────────────────────── │  (port 8000)        │
└──────────────────┘                              └─────────┬──────────┘
                                                              │
                                                              ▼
                                                   ┌────────────────────┐
                                                   │  LangGraph Agent    │
                                                   │  (Phase 3 graph)     │
                                                   └─────────┬──────────┘
                                                              │
                                                   ┌──────────┴──────────┐
                                                   ▼                     ▼
                                          ┌─────────────────┐  ┌──────────────┐
                                          │  FAISS Vectorstore│  │  Groq LLM     │
                                          │  (Phase 1)         │  │  (Phase 2)    │
                                          └─────────────────┘  └──────────────┘
```

Two separate processes, talking over HTTP. This mirrors real production
architecture: a stateless API server + a separate frontend client.

## Core FastAPI Concepts

### 1. Pydantic models = request/response contracts
```python
class ChatRequest(BaseModel):
    question: str = Field(..., min_length=1)
    session_id: Optional[str] = "default"
```
FastAPI auto-validates incoming JSON against this. Send `{"question": ""}` and
you get a 422 error automatically — no manual validation code needed.

---

### 2. Singleton pattern for expensive resources
```python
_vectorstore = None

def get_vectorstore():
    global _vectorstore
    if _vectorstore is None:
        _vectorstore = FAISS.load_local(...)   # only runs ONCE
    return _vectorstore
```
Loading the embedding model and FAISS index takes a few seconds. You do NOT
want to reload them on every request — load once, keep in memory, reuse.

---

### 3. UploadFile for file handling
```python
@app.post("/ingest")
async def ingest_document(file: UploadFile = File(...)):
    with open(file_path, "wb") as f:
        shutil.copyfileobj(file.file, f)
```
`UploadFile` streams the file to avoid loading huge files entirely into RAM.

---

### 4. Server-Sent Events (SSE) — the key streaming concept

```python
async def agent_event_stream(question, session_id):
    async for step in graph.astream({"messages": [...]}):
        # ... process each step
        yield f"data: {json.dumps(event)}\n\n"   # SSE format

@app.post("/chat/stream")
async def chat_stream(request: ChatRequest):
    return StreamingResponse(
        agent_event_stream(request.question, request.session_id),
        media_type="text/event-stream",
    )
```

**SSE format rules:**
- Every event: `data: {json}\n\n` (note the double newline — required!)
- `media_type="text/event-stream"` tells the browser/client this is a stream
- The client reads line by line and parses each `data:` line

**Why SSE over WebSockets here?**
SSE is one-directional (server → client), which is exactly what we need —
the client sends ONE request, then just listens. WebSockets are bidirectional
and add complexity we don't need for this use case.

---

### 5. `graph.astream()` vs `graph.stream()`
```python
# Sync version (Phase 3):
for step in graph.stream({"messages": [...]}):
    ...

# Async version (Phase 4, required inside FastAPI async endpoints):
async for step in graph.astream({"messages": [...]}):
    ...
```
FastAPI endpoints are `async def` — you must use the async LangGraph methods
(`ainvoke`, `astream`) to avoid blocking the event loop.

## Core Streamlit Concepts

### 1. Session State — Streamlit's memory
```python
if "messages" not in st.session_state:
    st.session_state.messages = []
```
**Critical to understand**: Streamlit reruns your ENTIRE script top-to-bottom
on every user interaction (button click, text input, etc). Without
`st.session_state`, your chat history would reset on every message.

---

### 2. st.chat_message + st.chat_input
```python
for msg in st.session_state.messages:
    with st.chat_message(msg["role"]):   # "user" or "assistant"
        st.markdown(msg["content"])

if question := st.chat_input("Ask something..."):
    # this block only runs when the user submits
    ...
```
These are purpose-built chat UI components — they render the chat bubble
styling automatically.

---

### 3. Reading an SSE stream from the client side
```python
response = requests.post(url, json=payload, stream=True)  # stream=True is key!

for line in response.iter_lines(decode_unicode=True):
    if line.startswith("data: "):
        event = json.loads(line[len("data: "):])
        # update UI based on event["type"]
```
`stream=True` in `requests.post()` prevents the client from waiting for the
entire response — it processes each line as it arrives, just like the
backend yields it.

---

### 4. st.empty() for live-updating placeholders
```python
status_placeholder = st.empty()
status_placeholder.markdown("🔧 Calling tool...")
# later...
status_placeholder.empty()   # clear it
answer_placeholder.markdown(final_answer)
```
`st.empty()` creates a placeholder you can overwrite repeatedly — this is
how the UI shows "Searching..." then replaces it with the final answer,
instead of stacking multiple messages.

## The Full Request Lifecycle (streaming mode)

```
1. User types question in st.chat_input()
2. Streamlit sends POST /chat/stream with {question, session_id}
3. FastAPI calls graph.astream() — starts the LangGraph agent loop
4. agent_node decides: needs search_knowledge_base
   -> yields SSE event {"type": "tool_call", "tool": "search_knowledge_base"}
   -> Streamlit shows: "🔧 Calling tool: search_knowledge_base..."
5. tools node executes the search
   -> yields SSE event {"type": "tool_result", ...}
   -> Streamlit shows: "✅ search_knowledge_base returned results, thinking..."
6. agent_node has enough info, generates final answer
   -> yields SSE event {"type": "answer", "content": "RAG stands for..."}
   -> Streamlit replaces status text with the actual answer
7. Stream ends -> yields {"type": "done"}
8. Streamlit saves the full exchange to st.session_state.messages
```

## Common Errors & Fixes

| Error | Cause | Fix |
|-------|-------|-----|
| `Connection refused` in Streamlit | Backend not running | Start `uvicorn main:app` first |
| CORS error in browser console | Missing CORS middleware | Already added in `main.py` — check `allow_origins` |
| Chat history disappears | Not using `st.session_state` | Always store messages in session state, never plain variables |
| SSE stream hangs / never completes | Missing `\n\n` after `data:` | Every SSE event needs `f"data: {json}\n\n"` exactly |
| `RuntimeError: This event loop is already running` | Mixing sync/async incorrectly | Use `astream`/`ainvoke` inside `async def` endpoints |
| File upload fails with 422 | Missing `python-multipart` | `pip install python-multipart` |
| First request very slow | Models loading on first call | This is expected — `@app.on_event("startup")` preloads them |
